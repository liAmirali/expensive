# 1. Overview

## 1.1 Product Vision

Expensive is a group-based expense sharing application that enables users to organize shared financial activities (trips, events, households, dormitories, projects, families) and accurately track who owes whom, with clarity, fairness, and minimal friction.

The product prioritizes:

- Financial transparency
- Flexible but consistent splitting logic
- Strong role-based collaboration
- Scalable and clean system architecture
- Localization-first design for the Iranian market
- International expansion readiness

---

## 1.2 Problem Statement

People frequently share expenses in:

- Trips
- Social gatherings
- Cafés and restaurants
- Dorm rooms
- Shared apartments
- Family environments
- Startup teams
- Long-term collaborative groups

Existing solutions often suffer from:

- Over-simplified split logic
- Overly complex accounting systems
- Weak access control
- Poor support for long-lived financial contexts
- Limited localization for Iranian users

Expensive aims to:

- Provide structured, flexible expense tracking
- Support both short-term and long-term financial collaboration
- Maintain clear and correct financial balances
- Provide strong access boundaries and data integrity
- Deliver a UX optimized for Iranian users while remaining globally adaptable

---

# 2. Target Audience & Market Positioning

## 2.1 Primary Target Audience (Phase 1)

Expensive is initially optimized for the **Iranian audience**, supporting:

- Iranian Toman (IRT)
- Toman display conventions (1 Toman = 10 IRR)
- Persian (Farsi) language
- Right-to-Left (RTL) interface
- Jalali calendar display
- Local settlement patterns (cash, card-to-card transfers, informal payments)

## 2.2 International Compatibility

The system must be architected to support:

- Multiple currencies (configurable per group)
- Locale-based number formatting
- Multiple languages (i18n-ready)
- Timezone-aware operations
- International deployment without architectural change

The core financial engine must remain **currency-agnostic and locale-independent**.

# 3. Core Concepts & Domain Model

This section defines canonical entities. These definitions must remain consistent across backend, API, and database.

---

# 3.1 User

## Definition

A registered individual with authentication credentials and profile information.

## Properties

- id (UUID)
- email (unique, required)
- password_hash
- full_name
- avatar_url (optional)
- phone_number (optional)
- default_currency
- preferred_locale
- created_at
- updated_at
- is_active (boolean)
- email_verified (boolean)

## Constraints

- Email must be unique.
- Password must meet defined security policy.
- Soft deletion preferred (is_active flag).

## Actions

- Register
- Login
- Logout
- Update profile
- Change password
- Deactivate account

---

# 3.2 Group

## Definition

A persistent financial container where users collaborate and create financial sub-contexts.

Examples:

- Family
- Dorm Room 314
- Travel Friends
- Startup Team

Groups are long-lived by default.

## Properties

- id (UUID)
- name
- description
- base_currency
- created_by (User)
- created_at
- archived_at (nullable)

## Constraints

- Must have exactly one Owner.
- Cannot be permanently deleted if financial records exist.
- Base currency must be defined at creation.

## Actions

- Create group
- Archive group
- Invite members
- Change member roles

---

# 3.3 Group Membership

Represents a user’s relationship to a Group.

## Properties

- id
- group_id
- user_id
- role (OWNER | ADMIN | MEMBER)
- joined_at
- invited_by
- status (PENDING | ACTIVE | REMOVED)

## Role Definitions

### Owner
- Full control
- Assign roles
- Archive group
- Remove members
- Promote/demote admins

### Admin
- Create/edit ledgers
- Invite/remove members
- Create/edit expenses
- Cannot remove Owner

### Member
- Participate in ledgers
- Create expenses
- Record settlements

Role-based permissions are sufficient for v1. Fine-grained scopes may be added later.

---

# 3.4 Ledger (Financial Sub-Context)

## Definition

A financial sub-container inside a Group that organizes a subset of members and related expenses.

Examples:

- Billy's Birthday
- Summer Trip 2026
- Monthly Rent - Room 314
- Family Shared Costs

Ledger is neutral and works for both short-lived and long-lived contexts.

## Properties

- id
- group_id
- name
- description
- visibility (PRIVATE_TO_PARTICIPANTS | VISIBLE_TO_GROUP)
- currency (inherits group by default)
- created_by
- created_at
- closed_at (nullable)

## Ledger Participants

- ledger_id
- user_id
- joined_at

## Constraints

- Must include at least one participant.
- Participants must belong to the group.
- Closed ledger cannot accept new expenses.
- Currency cannot change after first expense.

## Actions

- Create ledger
- Add/remove participants
- Close/reopen ledger
- View balances

---

# 3.5 Expense

## Definition

A financial transaction where one participant pays on behalf of one or more participants.

## Properties

- id
- ledger_id
- payer_id
- title
- description
- total_amount
- currency
- expense_date
- split_method (EQUAL | EXACT | PERCENTAGE | ITEMIZED) [Everything is EQUAL for now]
- created_by
- created_at
- updated_at
- is_voided (boolean)

## Constraints

- Payer must be ledger participant.
- total_amount > 0.
- Sum of splits must equal total_amount.
- One split method per expense.

---

# 3.6 Expense Split

Represents individual obligation per participant.

## Properties

- id
- expense_id
- user_id
- amount_owed

## Invariants

- Sum(amount_owed) = total_amount.
- Payer may owe part of expense.

---

# 3.7 Item (Optional Substructure)

Used for granular allocation.

## Properties

- id
- expense_id
- name
- amount
- quantity (default 1)

## Rules

If split_method = ITEMIZED:

- total_amount = sum(item.amount × quantity)
- Each item must be assigned
- Each item must be fully allocated
- Items cannot mix split modes

This avoids ambiguous edge cases.

---

# 3.8 Settlement

Represents real-world money transfer.

## Properties

- id
- ledger_id
- from_user
- to_user
- amount
- settlement_date
- note

## Constraints

- Must reduce outstanding balance.
- Cannot exceed debt.

# 4. Functional Requirements

---

# 4.1 User Management

- Email verification required.
- JWT authentication with refresh tokens.
- Profile update capability.
- Password change flow.
- Account deactivation.

---

# 4.2 Group Management

- Create group (creator becomes Owner).
- Invite via email or in-app.
- Accept/decline invitation.
- Role management by Owner.
- Archive group.

---

# 4.3 Ledger Management

- Create ledger within group.
- Define participants at creation.
- Add/remove participants.
- Close ledger.
- View net balances per ledger.

---

# 4.4 Expense Management

Supported Split Modes:

1. Equal split
2. Exact amount split
3. Percentage split
4. Itemized split

Editing expense must:
- Recalculate balances
- Be transactional
- Maintain data integrity

---

# 4.5 Balance Engine

System computes:

- Net balance per participant
- Optimized settlement graph

Requirements:

- Deterministic calculations
- Decimal precision
- No floating-point arithmetic
- Efficient for up to 200 expenses per ledger

---

# 5. Localization & Iranian Market Requirements

---

# 5.1 Currency

- Default: IRT.
- Toman display option (UI-only transformation).
- One currency per group.
- Stored using high-precision decimal.

---

# 5.3 RTL Support

---

# 5.4 Cultural Considerations

- Cash & card transfer friendly.

- Low bandwidth optimized UI.

# 7. Non-Functional Requirements

## Performance

- Expense creation < 300ms.
- Balance calculation < 500ms for 200 expenses.

## Scalability

- 100k users.
- 500-member groups supported.

## Security

- HTTPS required.
- Password hashing (Argon2 or bcrypt).
- Role-based authorization checks.

## Data Integrity

- All financial writes wrapped in DB transactions.
- No partial expense writes allowed.

## Auditability

- Soft deletion for expenses.
- Edit tracking (future enhancement).

---

# 8. System Constraints & Invariants

- Every expense must fully balance.
- No floating-point arithmetic.
- Ledger participants must belong to group.
- Removing user preserves history.
- Ledger currency locked after first expense.
- Toman conversion strictly UI-level.
