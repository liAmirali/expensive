# Scope Changes from v1.0

This document narrows v1.0 scope. v1.0 remains the canonical domain reference for anything not overridden below.

## Removed from v1.x

- **Email verification.** `emailVerified` field stays in schema for forward compatibility, but no verification flow ships. New accounts treated as if verified.
- **Account deactivation.** No user- or admin-facing deactivation. `isActive` defaults to `true` and is never toggled.
- **Group invitations (accept/decline).** No `PENDING` lifecycle. Adding a user to a group is immediate; `GroupMembership.status` is always `ACTIVE`.
- **Item-to-user assignment.** `Item` model unused in v1.x.
- **Multi-currency.** No currency field on `Group`, `Ledger`, or `Expense`. Everything is implicitly Toman.
- **IRT ↔ IRR transformation.** No conversion anywhere. Server stores Toman, UI displays Toman.
- **Locale switching.** `preferredLocale` field exists but is a stub; only `fa-IR` (Tehran) is supported.
- **Hard deletion of groups or ledgers.** Both are archive/close only.

## Lifecycle rules

- **Groups** can be archived but never deleted. Archived groups hidden from UI.
- **Ledgers** can be closed but never deleted. Closed ledgers hidden from UI.
- A ledger can only be closed if **all expenses are settled** (net balance = 0 for all participants).
- A group can only be archived if **all its ledgers are closed**.
- Closed ledgers in an archived group **cannot be reopened**.

## Split logic

- Only `EQUAL` split method ships in v1.x.
- Split calculation happens **server-side**. Client sends `totalAmount` + `payerId` + `participantIds`; server computes per-participant `amountOwed`.
- `EXACT`, `PERCENTAGE`, `ITEMIZED` enum values remain in schema for forward compatibility but are rejected by the API.

## Auth scope

- `register`, `login`, `refresh` only.
- No `logout` endpoint, no password change, no email verification, no deactivation.

# v1.0 Sections Still in Effect

Everything in `v1.0.md` not contradicted above remains binding:

- Core domain entities (§3)
- Ledger visibility (`PRIVATE_TO_PARTICIPANTS` | `VISIBLE_TO_GROUP`)
- Group roles (`OWNER` | `ADMIN` | `MEMBER`)
- Balance engine requirements (deterministic, decimal, no floats, ≤200 expenses)
- Settlement constraints (positive amount, cannot exceed debt)
- Non-functional requirements (performance, security, transactional writes)
- RTL + Persian numerals in UI
